Competitor Technology Intelligence Agent

제약사 경쟁사 분석을 자동화하는 멀티 에이전트 LangGraph 파이프라인. 재무제표·특허·뉴스 데이터를 멀티소스로 통합해 구조화된 인텔리전스 리포트를 자동 생성합니다. 기존 2주 이상 소요되던 수작업 분석을 완전 자동화, 2026년 6월 전사 배포 예정.

LangGraph 선정 이유

선형 체인(LCEL)으로 구현했을 때와 비교해 세 가지 문제가 구조적으로 해소됩니다.

① 정확성 — 호출부 실수로 틀린 값이 조용히 전파되는 문제
선형 체인에서는 호출부가 ch1, feedback, retries를 변수로 들고 다니며 직접 인자로 넣어줘야 합니다. 재시도 분기를 빠뜨리거나 실수로 이전 값을 넘기면 LLM은 오류 없이 실행되지만 틀린 컨텍스트로 챕터를 생성합니다. 이런 버그는 출력 텍스트를 읽기 전까지 감지되지 않습니다.

# 선형 체인 — ch2 재시도 중 ch1 최신값을 호출부가 보장해야 함
ch1 = generate_ch1(company)
ok1, feedback1 = review_ch1(ch1)
if not ok1:
    ch1 = generate_ch1(company, feedback=feedback1)
ch2 = generate_ch2(company, ch1)  # 이 ch1이 최신인지 호출부가 보장
ok2, feedback2 = review_ch2(ch1, ch2)
if not ok2:
    ch2 = generate_ch2(company, ch1, feedback=feedback2)
ch3 = generate_ch3(ch1, ch2, patents)  # 둘 다 최신인지 호출부가 보장

LangGraph에서는 state["ch1"]이 항상 마지막으로 쓰인 값입니다. 노드는 상태에서 꺼내 쓸 뿐이고, 어떤 노드도 다른 노드의 필드를 실수로 덮어쓸 수 없습니다.

② 확장성 — 필드 추가 시 모든 호출 지점을 수정해야 하는 문제
챕터가 늘면 선형 체인의 호출부에 chX, chX_ok, chX_feedback, chX_retries가 챕터 수만큼 쌓이고, 필드 하나가 추가될 때마다 그 값을 받아서 넘기는 모든 호출 지점을 찾아 수정해야 합니다. LangGraph는 AnalysisState 스키마에 필드 하나를 추가하면 끝입니다. 이후 어느 노드든 state["new_field"]로 꺼내면 되고, 다른 노드는 건드리지 않아도 됩니다.

③ 관찰 가능성 — 중간 상태 스냅샷이 없는 문제
선형 체인에서 실패하면 지역 변수는 마지막 값으로 덮어씌워지거나 사라집니다. "ch1이 재시도됐는지, 피드백이 뭐였는지"를 사후에 알 수 없습니다. LangGraph는 실패 시점의 AnalysisState 전체가 구조화된 객체로 남습니다.

# 실패 시점 상태 덤프 — 어느 노드에서 왜 멈췄는지 한눈에 파악
{
  "ch1": "...",  "ch1_ok": False,
  "ch1_feedback": "재무 수치 누락",  "ch1_retries": 1,
  "ch2": "",  # 아직 생성 안 됨
}

모든 노드 경계에서 log_callback(단계 로그)과 check_cancel(중단 지점)이 호출되는 것도 이 구조 덕분입니다.

각 노드는 인자를 직접 주고받지 않고 공유 상태 객체 AnalysisState를 읽고 부분 갱신하며 통신합니다. 예: review_* 노드가 chX_ok(통과 여부)·chX_feedback을 써넣으면 라우터가 그 값을 읽어 다음 노드를 결정합니다. 검토 실패 시 재생성 ≤ MAX_RETRIES로 동일 챕터를 다시 만듭니다.

전처리 (그래프 외부) Generate Review Synthesize Finalize FAIL 재생성 루프
노드 (LangGraph)유형기능핵심 구현 (코드 근거)
generate_ch1
entry_point		 Generate Chapter 1 (Corporate & Financial Overview) 초안·수정안 생성 analyst.generate_chapter1(company, is_subsidiary, parent_company_kr, feedback, …) — 재시도 시 직전 ch1_feedback 주입

Problems & Solutions

⚠️
연관 특허가 제대로 추출되지 않는 문제
▼
원인 분석
특허 검색은 외부 특허 API에 {assignee, keyword, cpc}를 넘기는 키워드·출원인 기반 구조화 질의입니다(임베딩·벡터 유사도가 아닙니다). 따라서 누락의 근본 원인은 성격이 다른 두 종류의 recall 실패이며, 같은 처방으로 풀 수 없습니다.

① 기술명 불일치 — 입력 용어 ≠ 특허 기재 용어 (recall 문제)
• 명세서의 의도적 추상화: 출원인은 권리범위(claim scope)를 넓히려 상위 개념어·기능적 표현을 의도적으로 사용합니다. "GLP-1 작용제"로 검색해도 명세서엔 "인크레틴 모방 펩타이드"로 기재되며 — 이는 노이즈가 아니라 특허 문서의 구조적 특성입니다.
• 다국어 출원: 동일 기술이 EN/KO/ZH/JA로 각기 다른 용어로 출원돼, 입력 언어 하나로는 회수 범위가 좁습니다.
• 구조화 키워드 검색에선 "글자 그대로 매칭되는 검색어 집합을 얼마나 넓히느냐"가 recall을 좌우합니다.

② 출원인명 불일치 — 동일 개체의 표기 변이 (entity resolution 문제)
• 법인 표기 변이: 법적 접미사(Co., Ltd. / 주식회사), 약어("Therapeutics" → "TX"), 한·영·중·일 표기 차이.
• 출원 주체 ≠ 브랜드: 특허청은 출원 당시 출원인을 기록 — 모회사 브랜드가 아니라 자회사·지주사로 등록된 경우가 흔합니다 → 검색 전 정규화가 선행돼야 합니다.
해결 접근
1
다국어 계층 키워드 + 특허 어휘 확장 (① 대응) — _get_tiered_keywords()가 LLM(Gemini)으로 기술명을 EN/KO/ZH/JA 계층 키워드로 확장하고, _get_patent_vocabulary()가 기술×기업 도메인의 특허 전문 용어를 추가 생성합니다. 둘을 en_base + patent_vocab + expanded_techs 합집합으로 묶어, 입력 1개를 다국어·전문용어 N개의 검색어 집합으로 넓혀 recall을 확보합니다.
2
다중 출원인 변형 매칭 (② 대응 · normalizer 연계) — 정규화 단계(standardize_entities())가 기업명 하나를 출원인 표기 변형 리스트 company_assignees와 언어별 표기(company_ja·company_zh)로 확장합니다. 검색기는 (키워드 × 모든 출원인 변형) 조합을 순회하고, 자회사는 Tavily로 모회사까지 해소해 누락을 막습니다.
3
CPC 분류 기반 보충 검색 (① 핵심) — 1차 검색 결과의 특허 분류코드를 집계해 상위 3개를 추출(cpc_counter.most_common(3))하고, 그 CPC + 출원인으로 재검색합니다. 용어가 전혀 달라도 같은 기술 분류에 속한 특허를 회수하며 — 임베딩 없이 특허청의 분류체계(CPC)를 신호로 삼아 어휘 불일치를 우회한 것이 이 설계의 핵심입니다.
4
중복 제거 + 랭킹 — Family ID 그룹화(같은 특허 패밀리는 최신 1건 유지, relevance score는 max 승계) + 동일 제목 2차 제거 후, 특허 API가 반환한 relevance score 내림차순으로 Top-30을 선정합니다. 비영어 제목은 _translate_to_english()로 영문 정규화합니다(독일어 복합어까지 탐지). — 별도 reranker를 두지 않은 이유는, 병목이 정밀도 정렬이 아니라 recall(관련 특허 누락)이었기에 재정렬 대신 회수 단계(키워드·CPC·출원인 확장)에 집중했기 때문입니다.

단순 특허 목록으로는 의미있는 기술 분석이 어려운 문제
▼
원인 분석
① 표면적 특허 데이터의 한계 — 특허 번호와 제목만으로는 실제 비즈니스 임팩트나 기술의 성숙도를 판단하기 어렵습니다.
② 특허와 실제 비즈니스의 단절 — 특허로 확보했더라도 상업화되지 않은 기술인지, 실제 매출을 발생시킨 기술인지 확인하기 어렵습니다.
③ 기술 성숙도 평가의 부재 — 연구실 단계인지, 파일럿 단계인지, 상용화 완료 단계인지 명확한 구분이 없습니다.
해결 접근
1
특허-매출 교차 검증 — 특허 청구항(Claim)과 초록(Abstract)의 기술 내용을 실제 뉴스, 공시, 재무 데이터와 매핑해, 해당 특허가 실제 사업화되어 매출을 발생시킨 기술인지 판별합니다.
2
4-Dimension 기술 분석 프레임워크 도입 — 각 핵심 특허에 대해 (1) 해결하려던 문제, (2) 기술적 접근 방식, (3) 기대 효과, (4) 경쟁사 대비 차별점을 명시적으로 추출하고 분석합니다.
3
TRL(Technology Readiness Level) + CRL(Commercial Readiness Level) 이중 평가 — 단순 기술 성숙도뿐 아니라 상용화 준비도까지 함께 평가해, 파이프라인의 현재 위치와 상용화까지의 거리를 정량화합니다.
